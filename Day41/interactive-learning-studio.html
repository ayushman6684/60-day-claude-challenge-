Object-Oriented Programming in Java

A complete, self-contained tutorial on classes, objects, encapsulation, inheritance, polymorphism and abstraction — taught from first principles through to a working mini system you design yourself.

Prerequisites

  • Basic Java syntax: variables, data types, methods, and how to compile/run a .java file.
  • Comfort with if/for control flow — no prior OOP knowledge required.

Learning objectives

  • Design classes with fields, constructors, and methods, and understand how objects are created from them.
  • Apply encapsulation to protect internal state using access modifiers and accessor methods.
  • Build class hierarchies with inheritance, and correctly use super, overriding, and constructor chaining.
  • Use polymorphism and abstraction — abstract classes and interfaces — to write flexible, extensible code.
  • Combine all four pillars to design a small, realistic multi-class Java system.

MODULE 01 / 04

Foundations — Classes, Objects & Encapsulation

Difficulty: ●○○○ Foundational

Every Java program is built from classes and objects. A class is a template — it defines what data an object holds and what it can do. An object is a concrete thing built from that template, living in memory with its own values.

AnalogyThink of a class as an architect's blueprint for a house. The blueprint isn't a house you can live in — it's a plan. Every actual house built from it (object) has its own address, paint color, and furniture, but they all share the same room layout defined by the blueprint.

Anatomy of a class

public class Car {
    // fields (state)
    private String model;
    private int speed;

    // constructor
    public Car(String model) {
        this.model = model;
        this.speed = 0;
    }

    // methods (behaviour)
    public void accelerate(int amount) {
        speed += amount;
    }

    public int getSpeed() {
        return speed;
    }
}
One class → many independent objects, each with its own field values

Encapsulation: protecting state

Fields marked private can't be touched directly from outside the class. Instead, we expose controlled access through public methods — commonly getters and setters. This lets the class validate or restrict changes to its own data.

AnalogyAn ATM machine encapsulates your bank balance. You never reach into the vault and change the number yourself — you go through the machine's buttons (methods), and it decides whether your withdrawal is valid.
Public methods act as the only doorway into private state

Comparison: class vs. object

ClassObject
A template / type definition, written onceA concrete instance created with new
Exists at compile time, no memory for instance dataExists at runtime, occupies heap memory
Example: CarExample: Car civic = new Car("Civic");

Practical exercise

Try it

Write a BankAccount class with a private double balance field, a constructor that sets an initial balance, and public deposit(double amount) and getBalance() methods. Sketch it below (this is a thinking exercise — check your approach against the model answer).

public class BankAccount {
    private double balance;
    public BankAccount(double initial) { balance = initial; }
    public void deposit(double amount) { balance += amount; }
    public double getBalance() { return balance; }
}
Common misconception"A class and an object are the same thing." They're not — a class is never instantiated on its own; it's the plan. Also: making fields public "to make things easier" defeats encapsulation and is considered poor Java style outside of simple data carriers.

Key takeaways

  • A class defines structure (fields) and behaviour (methods); an object is a live instance of it.
  • private fields + public methods = encapsulation, Java's mechanism for information hiding.
  • Constructors initialize an object's state the moment it's created with new.
MODULE 02 / 04

Inheritance

Difficulty: ●●○○ Building up
🔒
Complete the Module 1 quiz to unlock Inheritance.

Inheritance lets one class (the subclass) acquire the fields and methods of another (the superclass) using extends. It models an "is-a" relationship: a Dog is an Animal.

AnalogyThink of a family tree. Children inherit traits from parents automatically (eye color, last name) but can also develop their own unique traits. A Dog inherits eat() and sleep() from Animal, but adds its own bark().
public class Animal {
    protected String name;
    public Animal(String name) { this.name = name; }
    public void eat() { System.out.println(name + " is eating"); }
}

public class Dog extends Animal {
    public Dog(String name) {
        super(name); // calls Animal's constructor
    }
    public void bark() { System.out.println(name + " says Woof!"); }

    // overriding: replace parent's behaviour
    @Override
    public void eat() { System.out.println(name + " wolfs down kibble"); }
}
A superclass at the top, subclasses below — each "is-a" Animal

Overriding vs. overloading

OverridingOverloading
Same method signature, redefined in a subclassSame method name, different parameter list, same class
Resolved at runtime (dynamic dispatch)Resolved at compile time
Requires inheritanceNo inheritance needed
@Override public void eat()void feed(); void feed(int qty);

Practical exercise

Try it

Create a Cat subclass of Animal that overrides eat() to print "<name> sips milk", and adds a new method scratch().

public class Cat extends Animal {
    public Cat(String name) { super(name); }
    @Override
    public void eat() { System.out.println(name + " sips milk"); }
    public void scratch() { System.out.println(name + " scratches the post"); }
}
Common misconception"Java supports multiple inheritance of classes." It doesn't — a class can only extends one superclass (to avoid the "diamond problem"). Multiple inheritance of behaviour is achieved through interfaces instead, covered in the next module.

Key takeaways

  • extends creates an is-a relationship and reuses code from the superclass.
  • super(...) calls the parent constructor; it must be the first line if used.
  • Overriding changes behaviour polymorphically; overloading just adds variants of a method name.
MODULE 03 / 04

Polymorphism & Abstraction

Difficulty: ●●●○ Advanced concepts
🔒
Complete the Module 2 quiz to unlock Polymorphism.

Polymorphism ("many forms") lets you treat different subclasses through a single, shared type — and have the correct, specific behaviour run automatically at runtime. Abstraction is how we define that shared type without fully implementing it, using abstract classes or interfaces.

AnalogyA universal TV remote has one "power" button (the shared interface), but pressing it triggers a different internal circuit depending on whether it's plugged into a Samsung or an LG TV. You don't need to know the brand — you just press the button and the right thing happens.
interface Shape {
    double area(); // no body — a contract
}

class Circle implements Shape {
    private double r;
    Circle(double r) { this.r = r; }
    public double area() { return Math.PI * r * r; }
}

class Square implements Shape {
    private double side;
    Square(double side) { this.side = side; }
    public double area() { return side * side; }
}

// polymorphism in action:
Shape[] shapes = { new Circle(2), new Square(3) };
for (Shape s : shapes) {
    System.out.println(s.area()); // correct area() runs for each
}
One reference type, many runtime implementations — dynamic dispatch

Abstract class vs. interface

Abstract classInterface
Can hold fields, constructors, and partially implemented methodsTraditionally only method signatures (Java 8+ allows default methods)
A subclass can extend only one abstract classA class can implements many interfaces
Use when subclasses share common code/stateUse to define a capability/contract across unrelated classes
abstract class Animal { abstract void sound(); }interface Flyable { void fly(); }

Practical exercise

Try it

Add a Triangle class implementing Shape, with a constructor taking base and height, computing area() as 0.5 * base * height.

class Triangle implements Shape {
    private double base, height;
    Triangle(double base, double height) {
        this.base = base; this.height = height;
    }
    public double area() { return 0.5 * base * height; }
}
Common misconception"Polymorphism just means method overloading." Overloading is compile-time convenience; true polymorphism is about a single reference type (interface or superclass) dispatching to different runtime behaviour — that's what makes designs extensible without if/else chains on type.

Key takeaways

  • Polymorphism lets code depend on an abstraction (interface/superclass) instead of concrete classes.
  • Interfaces define "can-do" capabilities; abstract classes define "is-a" families with shared code.
  • Dynamic dispatch means the JVM picks the correct overridden method at runtime, based on the object's real type.
MODULE 04 / 04

Mastery — Designing a Real System

Difficulty: ●●●● Practical application
🔒
Complete the Module 3 quiz to unlock Mastery.

Real Java systems combine all four pillars at once. Let's design a tiny banking system: an abstract class Account holding shared state and behaviour, two concrete subclasses with different rules, and a Transactable interface so unrelated classes can share a capability.

interface Transactable {
    void withdraw(double amount);
}

abstract class Account implements Transactable {
    protected double balance;
    protected Account(double balance) { this.balance = balance; }
    public double getBalance() { return balance; }
    public abstract double interestRate();  // no shared answer
}

class SavingsAccount extends Account {
    SavingsAccount(double balance) { super(balance); }
    public double interestRate() { return 0.03; }
    public void withdraw(double amount) {
        if (amount <= balance) balance -= amount;
    }
}

class CurrentAccount extends Account {
    CurrentAccount(double balance) { super(balance); }
    public double interestRate() { return 0.0; }
    public void withdraw(double amount) {
        balance -= amount; // overdraft allowed
    }
}
Abstraction (interface) + inheritance (abstract class) + polymorphism (interestRate) + encapsulation (balance) — all four pillars, one design

Design principle: composition over inheritance

Not every relationship should be "is-a". If a class merely uses another (a Car "has-a" Engine), prefer holding a reference to that object (composition) instead of extending it. This keeps hierarchies shallow and avoids fragile, deeply-nested class trees — a widely used guideline in production Java codebases.

Inheritance ("is-a")Composition ("has-a")
class Dog extends Animalclass Car { private Engine engine; }
Tightly coupled to parent's implementationLoosely coupled — engine can be swapped freely
Good for genuine type hierarchiesGood for reusable, flexible behaviour

Practical exercise

Try it

Add a third account type, FixedDepositAccount, with a 5% interest rate and a withdraw() that always throws (no early withdrawal). Sketch the class.

class FixedDepositAccount extends Account {
    FixedDepositAccount(double balance) { super(balance); }
    public double interestRate() { return 0.05; }
    public void withdraw(double amount) {
        throw new UnsupportedOperationException("Locked until maturity");
    }
}
Common misconception"More inheritance = more reuse = better design." Deep hierarchies often become rigid and hard to change. Senior Java developers default to composition and use inheritance only when the "is-a" test genuinely holds and subclasses truly share behaviour.

Key takeaways

  • Real designs combine encapsulation, inheritance, polymorphism, and abstraction together.
  • Interfaces model capabilities; abstract classes model shared identity and partial implementation.
  • Prefer composition when the relationship is "has-a" rather than "is-a".

Summary notes

  • Encapsulation — hide state behind private fields and public methods.
  • Inheritance — reuse and specialize behaviour via extends and super.
  • Polymorphism — one type, many runtime behaviours, resolved via dynamic dispatch.
  • Abstraction — define contracts (interface) or partial templates (abstract class) without full implementation.
  • Composition over inheritance — favour "has-a" object references over deep class trees when there's no genuine "is-a" relationship.
